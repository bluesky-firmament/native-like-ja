# native-ja

文法的に正しい日本語と、ネイティブが実際に打つ日本語は別物だ。

「承知しました。確認いたします。」は間違っていない。
でも同僚へのチャットでこれが来たら、距離を感じる。あるいは怒ってるのかと思う。

native-ja は、文章を「日本語として正しい」から「日本語ネイティブが普段書く感じ」に寄せるスキルだ。


## humanizer-ja との違い

混同しやすいので先に整理しておく。

- **humanizer-ja** → AI が書いた文章から AI パターンを除去する。対象は書き言葉全般
- **native-ja** → 文章を日常的な「打ち言葉」に近づける。元が AI 生成かどうかは関係ない

humanizer-ja が「AI臭を消す」仕事なら、native-ja は「人と人の間で自然にやり取りできる文章にする」仕事。

併用する場合は、まず humanizer-ja で AI パターンを潰してから、native-ja で会話調に仕上げる。


## 何を扱うのか

native-ja が設計するのは「相互行為」の部分だ。
相手との距離、含み、同意の取り方、角の立てなさ。

具体的には 8 つのパターンを扱う。

| 番号 | 名前 | 一言 |
|---|---|---|
| NJ-01 | 句読点の温度シグナル | 「。」は冷たく読まれることがある |
| NJ-02 | 終助詞の応答型設計 | よ・ね・かな で相手の反応が変わる |
| NJ-03 | は／がのトピック制御 | 旧情報は「は」、新情報は述部側に |
| NJ-04 | 省略の安全範囲 | 省略していいところ、ダメなところ |
| NJ-05 | 談話標識の活用 | 「まあ」「で」「なんか」は潤滑油 |
| NJ-06 | ヘッジのフェイス保護 | 否定の前にワンクッション |
| NJ-07 | 慇懃無礼の検出 | 丁寧すぎると逆に冷たい |
| NJ-08 | 絵文字の非言語補助 | テキストだけだと意図が伝わりにくい場面の補助 |


## 場面で全部変わる

native-ja の核は「場面判定」にある。
同じ内容でも、友人チャットと職場チャットとビジネスメールでは書き方が全然違う。

| パラメータ | 友人チャット | 職場チャット | ビジネスメール |
|---|---|---|---|
| 句点「。」 | ほぼ使わない | 控えめ | 通常通り |
| 終助詞 | よ／ね／さ | ね／かな 程度 | 使わない |
| 省略 | 大量にOK | 中程度 | 最小限 |
| 敬語 | なし | 丁寧語 | 丁寧語＋尊敬語 |
| 絵文字 | 自由 | 最小限 | 使わない |

場面がわからないまま終助詞や省略を適用すると事故が起きる。
フォーマルな文書をくだけさせてしまったり、カジュアルな場面で硬すぎたり。
場面情報がなければ、まず聞く。


## Before / After

### 職場チャットでの依頼

**Before:**
> ご確認のほど、よろしくお願い申し上げます。

**After:**
> ここ見てもらえる？

---

### 否定を伝える場面

**Before:**
> それは無理です。

**After:**
> それ、ちょっと厳しいかも。別案でもいい？

---

### 話題を変える

**Before:**
> 資料を確認しました。修正をお願いします。

**After:**
> 資料見たよ。で、ここだけ直せそう？

---

### 感謝を伝える

**Before:**
> ありがとう

**After:**
> ありがとう！助かった😊


## 使い方

エージェントにスキルを読み込ませ、対象テキストと場面情報を渡す。

```
native-ja を使って、以下の文章を調整してください。
場面：職場の同僚へのチャット

（テキスト）
```

出力は以下の形式になる。

1. 場面判定とパラメータ設定
2. 改訂案
3. 変更点のまとめ


## 気をつけること

**過剰適用は「ネイティブごっこ」になる。**
全パターンを一度に詰め込むと不自然さが増す。
自然な文脈から生まれない談話標識やヘッジは入れない。

**省略しすぎると誤解を生む。**
主語が変わったのに省略したままだと、誰の話か分からなくなる。
「省略していい条件」と「してはいけない条件」の判定が大事。

**丁寧さは場面次第。**
友人チャットで丁寧すぎると壁を感じる。
ビジネスメールでくだけすぎると信頼を失う。
正解はひとつじゃない。場面ごとに変わる。
